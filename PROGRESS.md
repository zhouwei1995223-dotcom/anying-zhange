- Replaced actor center-only movement blocking with a small foot-radius check.

- Corrected movement progress tracking: only treat an actor as stuck when its own position barely changes, not when it temporarily fails to get closer to a moving target.
- Stuck actors now try a short retreat point only after real no-movement blocking, then choose a general bypass point if needed.
- Runtime actors are sorted by world Y each frame, so lower-screen actors draw above upper-screen actors.
- Added a quantized coordinate stuck check: while moving outside attack state, if the actor stays in the same small coordinate bucket for `0.5s`, it triggers retreat/bypass.
- Added actor separation checks so the hero and monsters cannot move/spawn/wander onto the same foot position.
- Hero targeting now refreshes to the nearest valid monster every frame, so nearby monsters encountered while walking are attacked first.
- Fixed tree/building edge sticking: bypass targets now stay active until reached or timed out instead of being cleared on the first tiny movement.
- Escape target search now checks multiple retreat/bypass radii and rejects points occupied by other actors, giving actors a better chance to leave tree and wall corners.
- Hero/monster targeting now checks `.blk` reachability within `900px`, so actors stop chasing enemies across rivers or outside walls when no path exists.
- Added target failure cooldown: if the hero chases one monster for `2.2s` without meaningful distance progress, that monster is ignored for `5s` so the hero can pick another target instead of looping at the same wall/tree edge.
- Added area-loop detection for target chasing: if the hero stays within an `80px` area for `2.2s` while chasing the same monster, that monster is also ignored for `5s`.
- Changed movement priority from "slide along obstacle first" to "A* around obstacle first"; slide movement is now only a fallback after direct movement and `.blk` path movement both fail.

## 2026-06-28 No-Code Authoring Requirement

- User requirement: the user is not expected to edit code. Finished gameplay systems must expose the useful settings in an editor-facing form.
- Old game reference: old server `.txt` tables are effectively game-design spreadsheets. They let designers change monster AI, map scenes, drops, and other values without touching source code.
- Remake direction: each confirmed feature should end with a simple Cocos authoring workflow:
  - select a node or config asset.
  - edit readable fields in the Inspector.
  - save the scene/config.
  - press Preview and see the result.
- Acceptance rule for future features: a feature is not complete if only Codex can change its important values through TypeScript. Important values must be exposed in the Inspector, a JSON config, or a simple editor tool.
- Implementation rule: start with the simplest editable surface that works. Use Cocos Inspector fields first; build custom editor panels only when Inspector lists become too hard to manage.

## 2026-06-29 Collaboration Rule

- Before changing code for a reported issue, first give a clear technical judgment: likely cause, proposed fix, risk, and test scope.
- The user knows the old game behavior well but is new to code, so implementation should follow discussion instead of silently guessing.
- Default workflow: diagnose briefly, state the smallest safe test, then implement after the direction is accepted.
